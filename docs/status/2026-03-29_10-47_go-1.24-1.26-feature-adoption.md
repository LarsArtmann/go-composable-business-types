## a) FULLY DONE

### 1. Monorepo Evaluation — REJECTED (Correct Decision)

The current single-module structure with subpackages is the ideal architecture. The packages form a deep dependency chain (`datapoint` → `actor` → `enums` → `id` → `types`) where splitting into separate modules would create versioning hell with zero benefit. Go workspaces are designed for independent modules with separate release cycles — the exact opposite of this library's tightly-coupled design.

### 2. `iter.Seq`/`Seq2` Iterator Methods — DONE

Added range-over-func iterators to two packages:

| Package     | Method            | Type                            | Purpose             |
| ----------- | ----------------- | ------------------------------- | ------------------- |
| `actor`     | `All()`           | `iter.Seq2[int, ActorEntry[T]]` | Index-value pairs   |
| `actor`     | `Entries()`       | `iter.Seq[ActorEntry[T]]`       | Values only         |
| `datapoint` | `AllReferences()` | `iter.Seq[Reference[string]]`   | Reference iteration |
| `datapoint` | `AllCauses()`     | `iter.Seq[Cause[string]]`       | Cause iteration     |
| `datapoint` | `AllTags()`       | `iter.Seq2[string, string]`     | Tag key-value pairs |

All methods support early `break` and handle empty collections. 10 new test functions covering iteration, break behavior, and empty collections.

### 3. `errors.AsType[E]` Generic Helpers — DONE

Added 4 type-safe error extraction functions in `pkg/errors/errors.go`:

```go
func AsUnmarshalError(err error) (*UnmarshalError, bool)
func AsValidationError(err error) (*ValidationError, bool)
func AsRangeError(err error) (*RangeError, bool)
func AsScanError(err error) (*ScanError, bool)
```

Each wraps Go 1.26's `errors.AsType[E](err)`, replacing `errors.As(err, &target)` boilerplate. Comprehensive tests including wrapped error chains.

### 4. `omitzero` JSON Struct Tags — DONE

Replaced `omitempty` with `omitzero` (Go 1.24) across 3 files:

| File                     | Fields Changed                           |
| ------------------------ | ---------------------------------------- |
| `datapoint/datapoint.go` | `Reason`, `Tags`, `References`, `Causes` |
| `datapoint/reference.go` | `Version`, `Tags`                        |
| `datapoint/cause.go`     | `Trace`                                  |

`omitzero` is semantically more precise: it omits fields that are the zero value for their type, rather than relying on `omitempty`'s "empty" heuristic (which treats `0`, `""`, `nil`, `false` as "empty").

### 5. `vcs.modified` Build Setting — DONE

Replaced the entire `isGitDirty()` function in `version/version.go` with a single line:

```go
case "vcs.modified":
    Dirty = setting.Value == "true"
```

**Removed:** `context`, `os/exec`, `strconv`, `time` imports, 5-second timeout, shell-out to `git diff --stat`, string parsing logic.  
**Result:** Zero-cost detection using Go's built-in VCS build info (available since Go 1.18, `vcs.modified` since Go 1.24).

## c) NOT STARTED

### 1. `encoding.TextAppender` / `encoding.BinaryAppender` Implementations

Go 1.24 introduced `encoding.TextAppender` and `encoding.BinaryAppender` interfaces. Types that already implement `encoding.TextMarshaler` (like `NanoID`, `Locale`, `Timestamp`) could gain zero-alloc `AppendText` methods:

```go
func (n NanoID) AppendText(b []byte) ([]byte, error) {
    return append(b, n.String()...), nil
}
```

This avoids allocation during encoding by appending directly to a caller-provided buffer.

### 2. `iter` Package Expansion

More types could benefit from iterator methods:

- `ActorChain.ByKind()` → could return `iter.Seq[ActorEntry[T]]` instead of slice
- `DataPoint` could get `All()` method yielding field names/values
- `Bitemporal` could iterate over time ranges

### 3. `go.mod` `ignore` Directive

Go 1.25 introduced the `ignore` directive for `go.mod` to exclude directories from `go test ./...` patterns. The `examples/` directories currently show as `[no test files]` in test output. Adding `ignore examples` would clean this up.

### 4. `sync.WaitGroup.Go` Adoption

Go 1.25 added `sync.WaitGroup.Go(func())` — a convenience method that combines `Add(1)` + `go func()`. Could replace patterns in test helpers or future concurrent code.

### 5. `testing/synctest` for Time-Dependent Tests

Go 1.25 graduated `testing/synctest` from experimental. Types like `Bitemporal`, `Timestamp`, and `DataPoint` have time-dependent behavior that could benefit from deterministic time control in tests.

### 6. Swiss Table Map Benefits (Automatic)

Go 1.24 switched to Swiss table map implementation — already active, but we haven't benchmarked to quantify the improvement.

### 7. `new()` with Expressions

Go 1.26 allows `new()` in expressions like `new(T).Field`. Not immediately applicable but useful in future API design.

2. **Add `iter` to more types** — `ActorChain.ByKind()` returns a slice allocation; an iterator variant would be more efficient. `Bitemporal` time-range iteration is a natural fit.

7. **`testing/synctest`** — Deterministic time control for `Bitemporal` and `DataPoint` tests.

8. **Example tests** — The `examples/` directory has no test files. Adding `Example*` functions would improve godoc.

### Lower Impact

## g) Top #1 Question I Cannot Figure Out Myself

**Should the `errors.AsType` wrapper functions in `pkg/errors/errors.go` exist at all?**

Since Go 1.26, any consumer can write:

```go
ue, ok := errors.AsType[*errors.UnmarshalError](err)
```

Our wrappers (`AsUnmarshalError`, `AsValidationError`, etc.) are trivial one-liners that call `errors.AsType` with a concrete type. The arguments for keeping them:

- **Encapsulation** — consumers don't need to know the concrete pointer types (`*UnmarshalError` vs `UnmarshalError`)
- **Discoverability** — `errors.As` is hard to find; named functions show up in godoc
- **Stability** — if we change the error types internally, wrappers provide a migration path

The arguments for removing them:

- **Thin wrappers** — they add zero logic over `errors.AsType`
- **Coupling** — they import `errors` from stdlib, creating a circular-ish documentation pattern
- **Generic native** — Go 1.26 users are expected to use `errors.AsType` directly

**Decision needed:** Keep, remove, or replace with a single generic function like `ExtractError[E error](err error) (E, bool)` that re-exports `errors.AsType` under our namespace?
